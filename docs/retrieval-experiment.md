# Retrieval experiment — MiniLM vs voyage-3

> Placeholder. The actual report is generated by
> `backend/scripts/retrieval_experiment.py`.

## How to run

Prerequisites:
- Qdrant container is up (`./manage.sh start` works).
- `VOYAGE_API_KEY` is set in `backend/.env` (get a key at
  <https://www.voyageai.com/>; the experiment costs a few cents).

Then:

```bash
cd backend
uv run python scripts/retrieval_experiment.py
```

The script will:

1. Drop and recreate two Qdrant collections (one per backend) so a
   re-run is idempotent.
2. Embed a fixed corpus (10 short docs, 8 tech + 2 off-topic controls)
   with both `sentence-transformers/all-MiniLM-L6-v2` and `voyage-3`.
3. Run 10 hand-labelled queries against both, at `top_k = 5`.
4. Compute recall@5 per query and the mean.
5. Overwrite this file with the results.

## Why this matters

`sentence-transformers/all-MiniLM-L6-v2` is the Phase 1 default — free,
local, 384-dim. `voyage-3` is the embeddings model Anthropic recommends
for use with Claude — API-served, 1024-dim, advertised to retrieve
better. The interview-talking-point version of this question is:
"how much does the API model actually help on documents like yours?"
This script answers that with measured numbers instead of vibes.

## Limitations

- Corpus is tiny (10 docs). Real workloads have hundreds to millions.
- Queries are crafted by the author — not user-derived.
- Single-relevant gold labels — multi-relevant requires LLM-as-judge
  or human pairwise judgement, which is its own bigger project.
- recall@5 with one relevant doc collapses to binary "did it appear in
  the top 5?" — fine as a first-look, weak for fine-grained tuning.

Treat the numbers as **signal, not verdict**. If you're picking an
embedding model for production, repeat the experiment on a slice of
your real data with real user queries.
