# GraphRAG experiment — does the graph help?

## How to run

Prerequisites:
- Qdrant + Neo4j up (`./manage.sh start`).
- `ANTHROPIC_API_KEY` in `backend/.env` (the experiment makes real
  Claude calls for question-side entity extraction).

```bash
cd backend
uv run python scripts/graph_experiment.py
```

The script:

1. Drops + recreates the `papermind_minilm` Qdrant collection.
2. Re-indexes a 10-doc corpus designed for graph value (entities that
   connect via curated edges, so the right doc for a question often
   isn't the one whose text matches the question lexically).
3. Plants 9 entities + 6 edges directly in Neo4j — **bypassing
   extraction** so the graph is ground truth relative to what the
   embedding alone can find.
4. Runs each of 10 queries through both modes and computes recall@5
   per query and overall.
5. Overwrites this file with the table.

## Why this matters

`docs/retrieval-experiment.md` answers "which embedding is better?".
This script answers a different question: **"given the same embedding,
how much does adding the knowledge graph help retrieval?"**

The hypothesis: for questions phrased around an entity whose chunk
doesn't repeat the entity's name, vector search misses but graph
augmentation finds it via the 1-hop neighbour edge.

## Limitations of this corpus

- **Tiny** (10 docs). Real workloads have 100s–millions, and graph value
  scales with corpus size + entity density.
- **Hand-labelled gold, one doc per query.** Multi-relevant labels
  require LLM-as-judge or human pairwise judgements — own project.
- **Graph is seeded, not extracted.** That isolates retrieval from
  extraction quality — useful here but means production results will
  be lower if extraction is noisy.
- **Production path differs.** In `/ask?use_graph=true` we DO use
  Claude-extracted entities (because we don't know the user's question
  in advance). This script measures the upper bound where the graph
  is perfect.

Treat the numbers as **directional signal**, not a verdict on
GraphRAG generally.
